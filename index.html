Section 2

Versioning

CSS is a living standard. CSS3 and HTML5 was the end of version numbers.

Context

Don't think in versions

CSS3 was the last numbered release. Modern CSS ships as specs agreed upon by the WC3 community, and implemented by browsers on their own schedule.

Modules however can be in different versions, such as subgrid is a level 2 feature in the CSS Grid module. But the key point is that there is no global version number to track. And you don't think of versions when it comes to HTML and CSS. Instead you think in…

Availability

Baseline badges

Not everywhere yet

Supported in some but not all major browsers. Experiment with fallbacks — not ready for production without them.

Ready, but recent

Supported in current major browsers, but older versions may still lag behind.

Safe default

Supported long enough across major browsers that most teams can use it without hesitation.

Section 3

Layout & Control

Two rules to hold in your head for the rest of the talk.

1D vs 2D · cascade priority stack

Big picture

Five layout algorithms

CSS isn't one layout system — it's five. The same property can behave differently depending on which algorithm is active. Most "why doesn't this work?" moments come from mixing up their rules.

The five

  • Flow — block & inline (the default)
  • Positioned — absolute / fixed / sticky
  • Table — tabular data, columns size to content
  • Flex — one-directional layout
  • Grid — two-dimensional layout

History

Evolution

Layout rule

Grid vs Flexbox

If you are laying things out in one direction, use Flexbox. If you are laying things out in two directions, use Grid.

Another lens: if you care deeply about track sizing and alignment, Grid gives you the right level of structure. If you care about flow and distribution, Flexbox is sufficient.

Rule of thumb

  • One axis → Flexbox
  • Two axes → Grid
  • Tabular data → Table

Control rule

The cascade priority stack

Why it matters

  • Most CSS conflicts are predictable
  • @layer gives you explicit control
  • Specificity is not the first tool to reach for

Section 5

Flexbox

One direction, practical alignment, and patterns you use every week.

Flow, alignment, distribution

Core idea

Flexbox is for one direction

Flexbox shines when you care about distribution and alignment along a single axis.

Where it wins

  • Navigation bars
  • Toolbars & button groups
  • Vertical stacks
  • Centering content

Key concept

Hypothetical size

flex-basis vs width

  • flex-basis wins over width when both are set
  • flex-basis works in the flex direction (height in column mode)
  • flex-basis: 0 = ignore content, divide by grow factors

Shorthand

The flex shorthand, decoded

Key difference

  • flex: 1 — items start from zero, equal widths
  • flex: auto — items start from content, share extra
  • flex: none — item is rigid

In practice

Proportional grow

How it works

  • flex-basis = minimum comfortable width
  • flex-grow divides extra space proportionally (1 : 3 : 1)
  • flex-wrap lets items wrap instead of being crushed
  • Fluid & responsive — no media queries needed

Practical

Patterns worth knowing

Watch out

  • margin-left: auto pushes items apart
  • min-width: 0 prevents overflow
  • gap replaces .item + .item hacks

@layer

Explicit cascade control

The idea

  • Declare layer order once
  • Later layers beat earlier layers
  • Regardless of specificity
  • Unlayered CSS beats all layers

The problem

Without @layer

Fragile

  • Same specificity — source order decides
  • Reorder imports → different winner
  • Add one selector → specificity war
  • Add !important → worse

The fix

With @layer

Predictable

  • Utilities layer beats components — always
  • Import order doesn't matter
  • Specificity doesn't matter
  • This is what Tailwind v4 does under the hood
  • Unlayered CSS (your DS) still beats all layers

Section 7

Scoping & Organization

Where does your CSS go? globals.css, component files, utility classes — and why styles leak.

Where does your CSS go?

The problem

Styles leak by default

Why it happens

  • CSS is global by default
  • Same class name = collision
  • Source order decides the winner
  • Importing a component can break another

The split

globals.css vs components

Component CSS

Approach 1

BEM — naming convention

Trade-offs

  • No tooling needed — works everywhere
  • Scoping through discipline
  • Verbose class names
  • Nothing stops you from breaking the convention

Approach 2

CSS Modules — tooling

Trade-offs

  • Write simple class names — tooling scopes them
  • Collisions are impossible
  • Requires a build step
  • Composing styles across files takes more work

Approach 3

Tailwind — utility classes

Trade-offs

  • No scoping problem — no custom classes
  • Design constraints built in
  • Verbose markup
  • Custom designs need @apply or config

Modern CSS Worth Knowing

Three features that meaningfully change how you build interfaces. All Baseline widely available.

Useful now, not just interesting

Feature

:has() — the parent selector

Why it matters

  • Parent-aware styling — no JS needed
  • Style wrappers based on children
  • Works on top of design systems you don't own

Feature

Container queries

Why it matters

  • Responsive to the component, not the viewport
  • Same component adapts to different contexts
  • Components respond to their container

Feature

:focus-visible

Why it matters

  • Focus rings only for keyboard users
  • Cleaner pointer interactions
  • Accessibility without visual noise
